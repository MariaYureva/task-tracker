# История разработки

Заметка работы над тестовым заданием - трекером рабочих задач для МИС.
Писался по ходу разработки, отражает ключевые решения и причины, по которым
проект устроен именно так.

## Что за задача

API-only приложение на Ruby on Rails: задачи могут быть разовыми и
повторяющимися (ежедневно, по чётным/нечётным дням, ежемесячно, по списку
конкретных дат), с тегами (включая защищённые системные), с независимым
статусом выполнения у каждого экземпляра, с возможностью переносить,
отменять и переопределять отдельные дни серии, а также редактировать серию
целиком, только один день или «этот и все последующие».

## Ключевое архитектурное решение: виртуальные вхождения

Главный вопрос задания - как хранить повторяющиеся задачи, чтобы серия из
«каждый день в течение года» не превращалась в 365 строк в базе и чтобы при
этом «проблема бесконечности» (серия без даты окончания) не ломала систему.

Решение - хранить правило, а не его развёртку. Задача (`Task`) описывает
серию: тип повторения, интервал, дату начала, опциональную дату окончания.
Конкретные вхождения на запрошенный диапазон дат вычисляются на лету чистой
функцией (`Occurrences::Expander`) - без обращения к базе и без сохранения.

В базу попадает только то, что отклонилось от правила: одна строка
`task_exceptions` на день, который перенесли, отменили, отметили выполненным
или переопределили. Ключ такого исключения - пара `(task_id, original_date)`,
то есть «какой день серии был изменён». Это позволяет:
- держать в базе минимум данных;
- сохранять историю выполненных дней даже после того, как правило изменилось
  и больше их не порождает;
- обязательный диапазон в запросе вхождений (и ограничение его длины - не
  более 366 дней) защищает от попытки развернуть бесконечную серию.

Статус выполнения (`pending / in_progress / completed / cancelled`) живёт на
вхождении (через исключение), а не на задаче - потому что каждый день
серии выполняется независимо. У самой задачи отдельный жизненный цикл
(`active / archived`): архивирование не теряет историю, в отличие от полного
удаления.

Календарные ловушки обрабатываются явно: ежемесячная задача на 31-е число
пропускает месяцы, где такого числа нет; 29 февраля не порождается в
невисокосные годы. Это покрыто отдельными тестами в `Recurrence edge cases`.

## Теги

Теги уникальны без учёта регистра (индекс по `lower(name)`). Системные теги
(`отчётность`, `операции`, `звонок`) создаются сидами, помечены `system: true`
и защищены от переименования и удаления на уровне модели и контроллера.

## Идентификация

Полноценная аутентификация не входила в задание, поэтому текущий пользователь
определяется по заголовку `X-User-Id`. Контроллеры наследуются от общего
`BaseController`, который подключает `Authenticatable` и централизованно
обрабатывает ошибки (404, 422, 400, 409).

## Развёртывание

Проект упакован в Docker: `Dockerfile` (Ruby 3.4.1) + `docker-compose.yml`
(приложение + PostgreSQL 16). Поднимается одной командой `docker compose up`.
При первом запуске entrypoint дожидается базы, создаёт её, прогоняет миграции
и сиды. Документация API доступна через Swagger UI на `/api-docs`.

## Тестирование

Покрытие на RSpec: юнит-тесты модели и чистого «расширителя» серий
(`Occurrences::Expander`), плюс request-тесты на все эндпоинты. Итоговый
прогон - 73 примера, 0 падений. Один тест на конфликт при переносе помечен
`pending`: его ожидаемое поведение противоречит соседнему тесту переноса
(перенос на штатный день серии в одном случае ожидается как успех, в другом
как конфликт — по содержимому эти случаи неотличимы), поэтому реализована
непротиворечивая семантика: перенос разрешён, конфликт возникает только при
занятости дня другим переносом.

## Ключевые запросы из переписки с LLM

В ходе работы LLM использовалась как ассистент - для проектирования
архитектуры, написания кода, настройки окружения и отладки. Ниже -
наиболее важные запросы в том виде, в каком они формулировались
(приведены близко к оригиналу, сокращённо).

### Проектирование и архитектура

- «Есть тестовое на Rails: трекер задач для МИС. Задачи могут повторяться
  (ежедневно, по чётным/нечётным, ежемесячно, по списку дат), бывают
  разовыми. Как лучше хранить повторяющиеся задачи в базе, чтобы серия
  "каждый день" не превращалась в сотни строк?»
- «Если серия бесконечная (без даты окончания), как не словить проблему
  бесконечности при выдаче вхождений? Нужно ли заставлять клиента всегда
  передавать диапазон дат?»
- «Где правильнее хранить статус выполнения, на самой задаче или на
  конкретном экземпляре дня? У каждого дня серии статус свой.»
- «Предложи модель, при которой в базу пишется только отклонение от правила, а обычные вхождения
  считаются сразу же. Как назвать таблицу и какой у неё ключ?»
- «Нужно три режима редактирования серии: всю целиком, только один день, и
  этот день со всеми последующими. Как это спроектировать на уровне сервиса?»

### Написание кода

- «Напиши чистый сервис-расширитель (Expander), который по правилу задачи и
  диапазону дат возвращает список дат вхождений, без обращения к БД.»
- «Сделай сериализаторы для задачи и вхождения, у вхождения нужен составной
  id вида task_id:original_date, статус, признак exception.»
- «Реализуй защиту системных тегов от изменения и удаления на уровне
  модели и контроллера.»
- «Нужен BaseController для API с подключением аутентификации по заголовку
  X-User-Id и централизованной обработкой ошибок (404, 422, 400, 409).»
- «Сгенерируй swagger.yaml для всех эндпоинтов в развёрнутом формате.»

### Docker и окружение

- «При первом запуске нужно дождаться готовности базы, создать её, прогнать
  миграции и сиды, как сделать entrypoint?»

### Отладка (по ходу запуска и тестов)

- «При старте контейнера падает uninitialized constant Api, почему?»
  (инициализатор inflections с акронимом API ломал автозагрузку
  Zeitwerk).
- «Puma падает на tmp/pids/server.pid, No such file or directory.»
- «NameError: uninitialized constant Occurrences::Occurrence::DEFAULT_STATUS
  как правильно объявить константу внутри Struct?»
- «Swagger UI пишет Failed to load API definition, в swagger.yaml ошибка
  на строке N, как проверить YAML на валидность?»
- «При прогоне RSpec все request-тесты падают с 403 Forbidden и HTML вместо
  JSON, хотя через curl всё работает. Прямой запрос в консоли даёт 200, почему?» (middleware host authorization резал запросы с
  хоста www.example.com, который использует RSpec).
- «uninitialized constant Shoulda при загрузке спеков, как подключить
  shoulda-matchers?»
- «Тест ждёт change(Task, :count).by(1), но получает by(2). Сервис в консоли
  создаёт ровно одну задачу. Почему через тест выходит две?» (ленивый let материализовался внутри блока expect).

### Чему научилась по ходу

Отдельно стоит отметить, что значительная часть запросов касалась не
написания кода, а понимания: почему контейнер ведёт себя так, как читать
трейс ошибки, чем отличается окружение test от development, как устроена
автозагрузка в Rails. LLM здесь работала как наставник, объясняющий причины,
а не только дающий готовые команды.
